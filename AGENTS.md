## ⭐ Golden rules (read first)

1. **Write your feature code in `src/modules/<feature>/`** and register the module in
   `src/modules/modules.module.ts`. See [`src/modules/README.md`](./src/modules/README.md).
2. **Never edit the generated files** — they are overwritten by the generator:
   - `src/generated.module.ts` · `src/generated.bootstrap.ts` · `src/generated.instrumentation.ts`
3. **Shared, cross-cutting code goes in `src/common/`** (helpers, guards, decorators, interceptors,
   filters, pipes) — not duplicated inside features.
4. **Before you finish a task, run `npm run lint && npm test`** and make sure both pass.
5. **Never commit secrets.** `.env` is gitignored; document new variables in `.env.example` only.
6. **Match the existing conventions** (below). Prefer small, focused modules over large files.

## What this project is

A modular NestJS (v10) REST backend. It was assembled from composable "blocks" (database, cache,
auth, logging, queue, messaging, observability). Whatever was selected at generation time is already
wired and working; everything else left **zero trace**.

**Discover the active stack** (don't assume) by reading:
- `package.json` → installed dependencies reveal the DB/ORM, cache, auth, etc.
- `src/generated.module.ts` → the exact feature modules wired into the app.
- `.env.example` → every configuration variable the app expects.
- `docker-compose.yml` (if present) → the backing services (mongo, postgres, redis, …).

## Architecture — the four layers

| Layer | Files | Edit it? |
|-------|-------|----------|
| **⚙️ Generated** | `src/generated.module.ts`, `src/generated.bootstrap.ts`, `src/generated.instrumentation.ts` | ❌ **Never** — regenerated by the CLI |
| **🏗️ Framework infra** | `src/main.ts`, `src/app.module.ts`, `src/common/`, `src/config/`, `src/health/` | ⚠️ Rarely — production wiring is already done |
| **🧩 Block scaffolds** | `src/database/`, `src/cache/`, `src/auth/`, … (only the selected ones) | ✅ Extend / replace — these are working examples (e.g. the sample `Item` CRUD) |
| **✍️ Your code** | `src/modules/<feature>/` | ✅ **This is where you work** |

`app.module.ts` imports two aggregators: `GeneratedModule` (framework features, CLI-owned) and
`ModulesModule` (your features). You only ever touch `ModulesModule`.

## Hooks (if `.agent-hooks/` or `.githooks/` exists)

If this project was generated with hooks, they run automatically — honour them rather than working
around them. There are two **independent, self-contained** layers (they share no files):

- **Agent hooks** (`.agent-hooks/`) — `.claude/settings.json` wires
  [Claude Code hooks](https://code.claude.com/docs/en/hooks) that run *while you (the agent) work*.
  ⚙️ `.claude/settings.json` is **generated — never hand-edit** (re-run the generator to change which
  hooks are wired). Logic + config live in `.agent-hooks/` (`config.json` = behavior, deny-lists, and
  the `enabled` on/off map).
- **Git hooks** (`.githooks/`) — `.husky/{pre-commit,pre-push,commit-msg}` run *on commit/push* (for
  humans and any tool too); each calls the `.githooks/run-phase.mjs` dispatcher. Logic + config live in
  `.githooks/` (its own `config.json`).

What you may encounter, and what it means:
- A blocked edit to `.env`/`migrations/` or a blocked shell command → the `guard` agent hook. Don't
  bypass it; if it's genuinely needed, do it manually or adjust `.agent-hooks/config.json`.
- Feedback about a hardcoded secret after an edit → the `secrets` agent hook. Move the value to
  `.env` / `ConfigService`.
- A commit rejected for its message format → `commitlint`; for being on `main`/`master` →
  `no-commit-to-main`. A commit/push that runs prettier, gitleaks, `tsc`, tests, or a build → the
  corresponding `git-hooks` check. Fix the reported issue rather than using `--no-verify`.

**Local overrides:** to turn a hook off *just for yourself*, copy the layer's `config.local.example.json`
to `config.local.json` (gitignored) in the same folder — `.agent-hooks/` for agent hooks, `.githooks/`
for git hooks — and set e.g. `{ "enabled": { "test": false } }`. It's deep-merged over that folder's
`config.json`, so it never affects teammates.

## Where to write code

```
src/modules/
├── modules.module.ts          # register every feature module here
└── <feature>/                 # one folder per feature
    ├── <feature>.module.ts        # bundles controller + providers
    ├── <feature>.controller.ts    # HTTP routes (annotate for Swagger)
    ├── <feature>.service.ts       # business logic
    ├── dto/                       # request/response shapes (class-validator)
    ├── entities/                  # data models (Mongoose → schemas/)
    └── <feature>.service.spec.ts  # unit test, co-located
```

**To add a feature:** copy `src/modules/example/` (or `nest g resource modules/<name>`), rename the
classes, then add it to `modules.module.ts` `imports`. The `example/` feature is a working reference
— study it, then delete it when you no longer need it.

**Models / controllers / services** are not top-level folders — they live *inside the feature that
owns them*. Only genuinely shared code goes in `src/common/`.

## Conventions

- **Feature-based structure** — group by domain feature, never by technical type.
- **Validation** — every request body is a DTO class with `class-validator` decorators. The global
  `ValidationPipe` (in `main.ts`) strips unknown fields and transforms types automatically.
- **Swagger** — annotate controllers/DTOs (`@ApiTags`, `@ApiOperation`, `@ApiProperty`). Docs serve
  at `/docs`.
- **Errors** — throw NestJS `HttpException`s (e.g. `NotFoundException`). The global exception filter
  (`src/common/filters/`) renders one consistent JSON error shape — don't hand-roll error responses.
- **Config** — read configuration via `ConfigService` (typed keys from `src/config/app.config.ts`),
  not `process.env` directly. Add new variables to `env.validation.ts` and `.env.example`.
- **Tests** — co-locate `*.spec.ts` next to the code; keep unit tests offline (mock DB/cache/network).
- **Logging** — inject `LoggerService` (or NestJS `Logger`); it's already the app logger.

## Commands

```bash
npm install            # install dependencies
cp .env.example .env   # then fill in values
docker compose up -d   # start backing services (if docker-compose.yml exists)
npm run start:dev      # dev server with watch  → http://localhost:3000
npm run build          # compile to dist/
npm test               # unit tests
npm run test:e2e       # end-to-end tests
npm run lint           # eslint --fix
```

- Swagger UI: `http://localhost:3000/docs`
- Health check: `GET http://localhost:3000/health`
